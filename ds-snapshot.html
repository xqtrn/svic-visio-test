Design System

The single source of truth for how every screen, document and surface looks, reads and behaves. Built for institutional restraint — the reference is BlackRock, Macquarie and Goldman Sachs. If something decorates without carrying meaning, it does not belong here.

01 — Foundations

Principles

Five rules govern every decision. When in doubt, remove rather than add. The interface should read like a document from an investment bank, not a consumer app.
Restraint over decoration. Whitespace, hairlines and one accent do the work. No decorative gradients, no drop shadows on content, no neon, no illustration. The single sanctioned gradient is the navy confidential cover (§03).
One accent. A single blue carries every interactive cue. Status uses muted semantic pairs only. Never a second decorative accent.
Serif for display, sans for work. Headings, names and large numbers are serif; body, labels and controls are sans. The serif is what reads as "institutional".
Hairlines, not boxes. Structure is drawn with 1px lines on a quiet field, not with shadows or heavy borders. Corners are calm (4–6px), never bubbly.
Generous space. Air is the premium signal. Pad rooms, separate sections, let the page breathe.

04 — Foundations

Tokens

Every colour, radius and spacing value is a named token in /app.css — never a raw hex in a screen. Tokens run in three tiers: raw primitives alias to semantic roles, which components consume. Re-theming (light to dark) swaps the semantic layer only; nothing downstream changes.

The three tiers

Primitive
Raw values — #0A64BC, the grey ramp. Named by what they are; no component reads them directly.
Semantic
Roles — --accent, --ink, --panel, --line. The only layer a theme overrides.
Component
Usage — a button reads --accent, a card reads --panel. Components never see a primitive.

Dark theme

The whole system re-themes by setting data-theme="dark" on the root — semantic tokens flip and every component follows, because none of them hardcode a colour. Initialisation is pre-paint: /loader.js, loaded synchronously in the head of every page, applies the saved or OS theme before first render on pages marked <html data-theme-auto> (the token-complete surfaces: this doc, the gallery, e-sign); legacy pages pin light until their stylesheets are converged. A manual choice persists per device. Try it on this page:
One source. All tokens live in /app.css :root — colour primitives and roles plus the scales --sp-1..7 (4/8/12/16/20/26/40), --r-control/card/panel (4/6/8), --t-fast/med/slow (150/180/250ms) and --z-* layers. A screen never hardcodes a hex, radius or spacing — it reads a token, so one edit re-skins everything and a theme is just an override block.

08 — System

Layout contract

The numeric contract every SVIC screen obeys. What is not written here is forbidden. A new page is assembled from a skeleton, the container whitelist, the rhythm table and the type scale — it never invents a size. Enforcement: font sizes and containers are linted on every page (0ds) — new pages strict against the published scale, legacy pinned in design-scale-baseline.txt, which can only shrink; padding and gap follow the --sp-* guidance scale (not yet machine-enforced).

Containers — the whitelist

Class / tokenMax widthUse
.w-app · --w-app1280pxApplication screens, lists, dashboards — the default
.w-read · --w-read768pxReading column: articles, docs, legal
.w-form · --w-form560pxSingle-purpose forms (auth, request access)
.w-wide · --w-wide1400pxFull data tables of the marketplace class only
Side padding is part of the container: 26px desktop, 18px ≤640. Any other max-width on a page is a lint failure.
The chrome shares the content grid — nothing overhangs the content column. Every horizontal band — the top bar, the page content, any sticky control bar — sits in the same container. The top bar's inner row (.topbar-in) uses the same max-width and side padding as the content below it, so the burger and brand line up with the content's left edge and the right-side controls (search/social) line up with its right edge. The bar's controls sticking out past the content column is a bug, not a style — the left edge of the burger must equal the left edge of the first content block, and the right edge of the last control must equal the content's right edge.
On a skinned surface, align the chrome to the skin's grid — not the 1280 default. A skin surface (§21) may carry a narrower content grid than the 1280 default — e.g. the WP media skin centres its content at ~1160px. There the DS top bar must be aligned to that grid: measure the skin's content max-width (render it, read the left edge of the content column at a known viewport) and set .topbar-in to match — never leave the 1280 bar overhanging a 1160 column. The canonical 1280 stays for native DS pages; only the skinned chrome is re-fitted.

12 columns, 24px gutter (--grid-cols / --grid-gutter) inside a .w-app container. Hairline grids (cards, terms) build from gap:1px over the line colour. Columns per breakpoint: 12 desktop · collapse to 2-up ≤900 · single column ≤640.

Top-line preloader — mandatory on every page (R-11)

The 2px accent line pinned to the very top of the viewport, above the top bar is the global load indicator: track --accent-track, fill accent→--accent-mid→accent (both dark-aware), driven by --page-progress. Every page includes /loader.js (auto-runs: 8% at parse → 38% @140ms → 72% at DOMContentLoaded → 100% at load → fade 480ms). It is a fixed standalone bar (#ds-loader) on every page; [data-progress-host] is a rare escape hatch to host the line on another element instead. Programmatic milestones: dsLoaderSet(0..100) / ui.js setProgress. Own progress bars are forbidden; lint 0du enforces the include.

Page skeletons

Every new page starts from a skeleton in the gallery — App list · Article · Form · Auth gate · Data table. A page that does not start from a skeleton is a review failure (R-10). Skin surfaces (WP media, marketplace) keep their frozen grids — see §Skinning.

10 — System

Iconography

One line style. Inline SVG, 1.6–2px stroke, currentColor, 16–22px. No emoji, no filled glyphs, no second icon library. Icons inherit text colour and only carry the accent on interactive states. Pair an icon with a label wherever meaning could be ambiguous.

URLs & addressable state

Every distinct view and every meaningful piece of state has its own URL. A screen is never a dead end you can only reach by clicking — it is a place you can link to, bookmark, share and reopen exactly as it was. State that changes what is shown lives in the path or the query string, not only in memory.
/marketplaceadmin?tab=sell&complete=false&sortColumn=date&sortDirection=desc&deal=31753
/whatsapp/rawMessages
Tab, filter, sort and an open record are all URL state. The active tab, every applied filter, the sort column and direction, and any opened deal or message live in query params (?tab=sell&sortColumn=date&deal=31753) so the exact view is reproducible from the link alone.
Each sub-area has a clean path. Distinct surfaces get their own readable route (/whatsapp/rawMessages), not a hidden mode of one mega-page.
Deep links open in context. Loading a deal/record URL opens that item inside its list with the surrounding filters intact — never a bare modal with no way back to the list.

20 — Operating

Governance & enforcement

The design system is not advisory — it is the code, and it is enforced mechanically so it cannot erode across sessions.
One source of truth. All tokens and components live in /app.css + /ui.js. New screens inherit them; they never invent local colours, radii or fonts.
Pre-deploy lint. A build step rejects any new page that uses a gradient, an inline drop-shadow, an emoji, a second accent, native alert/confirm/prompt, or a radius over 8px.
Vision QA — pixel-match against this page, before every deploy. Always, before any deploy, render the change in a real browser, screenshot it, and compare the resulting pixels against the canonical rendering on this design-system page — this page is the reference. If they do not match (wrong icon weight, broken proportions, a colour or spacing drift), it is a regression: redo and re-shoot until the pixels match. Never ship on a code review alone — code can pass while the render is wrong; the rendered pixels are the contract.
Changing the system. A new component is added to /app.css once, documented here, then reused — never a one-off in a page.
Versioning. The system carries a version; material changes bump it and note the date.
Contribution. Need a component the system lacks? Add it to /app.css with all states, document it here, then use it — never a one-off in a page.
Deprecation. A retired pattern is marked here, kept working one release, then removed from /app.css in the next.
