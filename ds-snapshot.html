Design System

The single source of truth for how every screen, document and surface looks, reads and behaves. Built for institutional restraint — the reference is BlackRock, Macquarie and Goldman Sachs. If something decorates without carrying meaning, it does not belong here.

01 — Foundations

Principles

Five rules govern every decision. When in doubt, remove rather than add. The interface should read like a document from an investment bank, not a consumer app.
Restraint over decoration. Whitespace, hairlines and one accent do the work. No decorative gradients, no drop shadows on content, no neon, no illustration. The single sanctioned gradient is the navy confidential cover (§Colour).
One accent. A single blue carries every interactive cue. Status uses muted semantic pairs only. Never a second decorative accent.
Serif for display, sans for work. Headings, names and large numbers are serif; body, labels and controls are sans. The serif is what reads as "institutional".
Hairlines, not boxes. Structure is drawn with 1px lines on a quiet field, not with shadows or heavy borders. Corners are calm (4–6px), never bubbly.
Generous space. Air is the premium signal. Pad rooms, separate sections, let the page breathe.
The litmus. Place a screen next to blackrock.com or macquarie.com — if ours looks brighter, busier or denser, it has regressed.

04 — Foundations

Tokens

Every colour, radius and spacing value is a named token in /app.css — never a raw hex in a screen. Tokens run in three tiers: raw primitives alias to semantic roles, which components consume. Re-theming (light to dark) swaps the semantic layer only; nothing downstream changes.

The three tiers

Primitive
Raw values — #0A64BC, the grey ramp. Named by what they are; no component reads them directly.
Semantic
Roles — --accent, --ink, --panel, --line. The only layer a theme overrides.
Component
Usage — a button reads --accent, a card reads --panel. Components never see a primitive.

08 — System · normative

Layout contract

The numeric contract every SVIC screen obeys. What is not written here is forbidden. A new page is assembled from a skeleton, the container whitelist, the rhythm table and the type scale — it never invents a size. Enforcement: §Governance, R-2–R-3.

Containers — the whitelist

Class / tokenMax widthUse
.w-app · --w-app1280pxApplication screens, lists, dashboards — the default
.w-read · --w-read768pxReading column: articles, docs, legal
.w-form · --w-form560pxSingle-purpose forms (auth, request access)
.w-wide · --w-wide1400pxFull data tables of the marketplace class only
Side padding is part of the container: 26px desktop, 18px ≤640. Any other max-width on a page is a lint failure.
The chrome shares the content grid — nothing overhangs the content column. Every horizontal band — the top bar, the page content, any sticky control bar — sits in the same container. The top bar's inner row (.topbar-in) uses the same max-width and side padding as the content below it, so the burger and brand line up with the content's left edge and the right-side controls (search/social) line up with its right edge. The bar's controls sticking out past the content column is a bug, not a style — the left edge of the burger must equal the left edge of the first content block, and the right edge of the last control must equal the content's right edge.
On a skinned surface, align the chrome to the skin's grid — not the 1280 default. A skin surface (§Skinning) may carry a narrower content grid than the 1280 default — e.g. the WP media skin centres its content at ~1160px. There the DS top bar must be aligned to that grid: measure the skin's content max-width (render it, read the left edge of the content column at a known viewport) and set .topbar-in to match — never leave the 1280 bar overhanging a 1160 column. The canonical 1280 stays for native DS pages; only the skinned chrome is re-fitted.

Top-line preloader — mandatory on every page (R-11)

The 2px accent line pinned to the very top of the viewport, above the top bar is the global load indicator: track --accent-track, fill accent→--accent-mid→accent (both dark-aware), driven by --page-progress. Every page includes /loader.js (auto-runs: 8% at parse → 38% @140ms → 72% at DOMContentLoaded → 100% at load → fade 480ms). It is a fixed standalone bar (#ds-loader) on every page; [data-progress-host] is a rare escape hatch to host the line on another element instead. Programmatic milestones: dsLoaderSet(0..100) / ui.js setProgress, and startProgress() / doneProgress() for fetch-level work. No spinners thrown over content for page-level loads. Own progress bars are forbidden; lint 0du enforces the include.

09 — Identity

Iconography

One line style. Inline SVG, 1.6–2px stroke, currentColor, 16–22px. No emoji, no filled glyphs, no second icon library. Icons inherit text colour and only carry the accent on interactive states. Pair an icon with a label wherever meaning could be ambiguous.

Drawer anatomy — navy header (logo + close), a labelled nav list (pinned items such as Fund Admin first, the active item carries an accent left-rule and wash), and a footer with the user name, role and sign-out.
One trigger, one drawer. Never a second horizontal nav or a competing dropdown. On mobile the drawer is the whole navigation; on documentation pages a sticky left TOC is added. Principles of the public surface

One navigation
A single parsed menu tree feeds every chrome surface — visible bar, megas, drawer, footer. A second hand-typed menu is a defect, not a shortcut.
Calm chrome

Content-first media

Faces, not centres

Motion is utility
Video plays one-at-a-time, muted, looped, and yields to prefers-reduced-motion. Chrome motion exists to serve reading, never to decorate.
Depth by lines

20 — Operating

Governance & enforcement

The design system is not advisory — it is the code, and it is enforced mechanically so it cannot erode across sessions.
One source of truth. All tokens and components live in /app.css + /ui.js. New screens inherit them; they never invent local colours, radii or fonts.
Pre-deploy lint. A build step rejects any new page that uses a gradient, an inline drop-shadow, an emoji, a second accent, native alert/confirm/prompt, or a radius over 8px.
Vision QA — pixel-match against this page, before every deploy. Always, before any deploy, render the change in a real browser, screenshot it, and compare the resulting pixels against the canonical rendering on this design-system page — this page is the reference. If they do not match (wrong icon weight, broken proportions, a colour or spacing drift), it is a regression: correct the build and re-verify until the renders match. Never ship on a code review alone — code can pass while the render is wrong; the rendered pixels are the contract.
Adding a component — definition of done. Into /app.css / ui.js once — never a one-off in a page. Ships with every state (default / hover / focus / disabled / loading), both themes and mobile ≤640; keyboard + ARIA per §Accessibility; documented here with a since x.y tag, a live example in the gallery and a changelog line — one PR, and 0dt passes with zero new baseline entries.
Versioning. Semver-lite: removing or renaming a shipped class or token bumps the major; a new component, token or pattern bumps the minor; doc-text-only edits change the date, not the version. A bump lands in three places at once — the hero meta, the top changelog row and the §Resources footer stamp — and the drift audit (Step 0dr) fails the build if the hero and the top changelog row disagree. Token changes regenerate /tokens.json (scripts/build-design-tokens.js); a stale export also fails 0dr. «One release» in the deprecation rule below means one minor or major bump.
Deprecation. A retired pattern is marked here first — its §Components subheading carries deprecated x.y → use Z next to its «since» tag, and its rule in /app.css gets /* @deprecated use Z */ — kept working one release, then removed from /app.css in the next, with a changelog line. New pages never use a deprecated class (review; on the first real deprecation the class name joins the 0dt scan).
Documentation register. Guidance here states the rule and its reason — never the incident that produced it.

Public-site gates (2026-07-05). (a) Inline-JS parse gate — every script string injected into served HTML must parse standalone (node --check) before deploy: a slash-regex cooked through a template literal becomes // and silently kills the whole script (three bites in one day). (b) Numeric render-verify — a change is «done» only after a real-engine render (desktop 1440, mobile 390, WebKit/iOS) passes MEASURED probes (aspect, gaps, baselines) — §Public site P-40. (c) Video defects are judged on PLAYING frames, never on markup or a single poster frame.
